# TaskHub — Where Team Chaos Becomes Clarity

> "Half of a project's failures don't happen in the code — they happen in the communication gaps between the people writing it."

That's the problem TaskHub was built to solve.

## The Story Behind It
Every software team knows this feeling: a requirement gets passed down, and by the time it reaches the developer, it means something completely different than what the stakeholder intended. A tester finds a blocker with no clear owner. Two developers touch the same database schema and no one notices until it breaks.

**TaskHub** is an HCI-driven mobile platform designed to close that gap — giving every role in a dev team (Leader, Analyst, Developer, QA) a single source of truth for what's happening, who owns it, and what's blocking it.

## The Problems We Tackled
| Problem | Impact |
|---|---|
| Ambiguous Requirements | Stakeholder expectations vs. dev output misalignment |
| Testing Blockers | Undocumented obstacles stalling QA |
| DB Schema Conflicts | Developers overwriting each other's work |
| Fragmented Visibility | No unified view of task status across roles |

## The Solution
A role-based dashboard system — everyone sees exactly what they need, nothing more, nothing less:

| Role | What They Get |
|---|---|
| Team Leader | Requirements inbox, task assignment, full team visibility, SOS alerts |
| Business Analyst | Requirement file uploads & tracking |
| Software Developer | Task status updates, progress tracking |
| QA Lead | Verification workflow, testing notes |

## Key Screens
- **Get Started** — Pick your role, get your view
- **Leader Dashboard** — Requirements + live Team Task table with priority & progress bars
- **Create New Task** — Fast task creation with role assignment
- **Quick SOS Access** — Escalated blockers, front and center, before they become fires

## Our HCI Process
We didn't just design screens — we designed for real team friction:

1. **How Might We (HMW)** — Turned raw pain points into design opportunities
2. **Personas** — Built out all four roles to understand conflicting needs
3. **Usability Testing** — Real users, real prototype, real feedback
4. **Cross-team Evaluation** — Every team's design got reviewed by another team — no echo chambers, just honest critique
